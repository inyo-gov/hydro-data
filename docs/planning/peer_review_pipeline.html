# Peer Review Pipeline (Draft)

This workflow links the repository to a DOI in Zenodo and prepares a reproducible, GitHub-native submission path for PeerJ (and optionally JOSS).

## 1) Zenodo DOI (release-based)

- [ ] Link the GitHub repo to Zenodo (one-time setup in Zenodo UI).
- [ ] Add release metadata (title, description, version) in GitHub.
- [ ] Create a release tag for each water-year update (e.g., `v2025.0`).
- [ ] Verify Zenodo minted a DOI for the release.
- [ ] Record the DOI in `README.md` and the report front matter.

## 2) Repository metadata (recommended)

- [ ] Add `CITATION.cff` with project title, authors, and DOI.
- [ ] (Optional) Add `zenodo.json` for detailed metadata.
- [ ] Confirm license and versioning strategy (semantic or annual).

## 3) PeerJ submission path

- [ ] Create a `paper/` directory with:
  - [ ] `paper.md` (PeerJ-format manuscript in Markdown or Quarto)
  - [ ] `references.bib`
  - [ ] `figures/` (exported, final figures)
- [ ] Preferred submission formats: PDF, Word, LaTeX, or Markdown/HTML. JATS is not required unless requested by the journal.
- [ ] Export a PDF from Quarto for submission review.
- [ ] Add a reproducibility section pointing to:
  - [ ] This repo
  - [ ] The Zenodo DOI
  - [ ] Rendered `docs/` outputs
- [ ] Keep results modest and data-driven (trend summaries and QA/QC).

## 4) Optional JOSS/J0SE track (software method)

- [ ] If the repo is positioned as a toolchain, add:
  - [ ] `paper.md` (JOSS format)
  - [ ] `paper.bib`
- [ ] Open a JOSS/JOSE submission issue when ready.

## 5) Annual cycle (recommended)

- [ ] On WY completion:
  - [ ] Render Quarto site
  - [ ] Update outputs in `output/`
  - [ ] Tag release in GitHub
  - [ ] Confirm Zenodo DOI minted
  - [ ] Log summary in `planning/worklog.md`
